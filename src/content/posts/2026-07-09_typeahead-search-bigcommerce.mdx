## How do I add typeahead search to my BigCommerce store?

Type a product name into your BigCommerce search box and nothing happens until you hit enter - that's the default behavior. A shopper looking for canvas sneakers types the words, watches the box sit there blank, and assumes you don't carry what they want. They leave before the results page ever loads, and because the query was never submitted, that abandonment never shows up in your analytics as a search term you can investigate. Typeahead search closes that gap: an instant, as-you-type dropdown that shows matching products, collections, and content before the shopper even finishes a word. Nobi adds it to a BigCommerce storefront with one line of code, and on a comparable storefront it lifted conversion rate by 0.5 percentage points - here's how to get it running and measure the same lift on your own traffic.

## Why do shoppers leave a BigCommerce search box before they finish typing?

Shoppers expect results to appear the moment they start typing, but BigCommerce's default storefront search waits for a full query and an Enter press. That gap between expectation and behavior is where search abandonment starts. A shopper types "canvas sneaker," sees nothing happen, and assumes the store has nothing relevant, so they bail before the results page ever loads.

That blank-screen moment is the real problem. There's no feedback loop telling the shopper the store is even listening, let alone that it has matching products. On a phone, where every extra tap costs attention, a few seconds of silence reads as a dead end.

This kind of abandonment never shows up in your analytics as a zero-result query. The search was never submitted, so there's no logged term, no null-results report, no line item to investigate. It's invisible churn: the shopper left, but nothing in the data says why, or even that it happened.

The shoppers most likely to leave here are your best ones. Someone who's already typing a specific product name isn't browsing. They know what they want and expect the store to keep up. When it doesn't, they don't wait around.

## What is typeahead search, and why does it fix that drop-off?

Typeahead search (also called instant search, or predictive search) closes that gap directly. Instead of waiting for a shopper to finish typing and hit enter, it matches against product titles, SKUs, and collection names on every keystroke and shows results in a dropdown right away. The shopper never faces a blank screen wondering if the store carries what they want. They see proof, in real time, that the store is listening.

That's a different thing from semantic matching, which is about whether the results are the *right* ones. Typeahead is about speed and feedback: does something show up the instant you start typing "canvas sneak..."? Semantic matching is about relevance: does the store understand that "wide-leg cropped trouser" and "cropped wide pant" mean the same thing? A store can have fast typeahead with weak matching (quick results, wrong ones) or slow search with great matching (right results, too late). Both matter, and they solve different failure points. This guide comes back to that distinction later, because fixing one without the other only gets you halfway there.

For now, the part typeahead handles is the drop-off itself. Every shopper who leaves before submitting a query is invisible in your analytics: no logged term, no zero-result report, nothing to investigate. A responsive dropdown catches a share of those shoppers before they bail, because it gives them a reason to keep typing instead of assuming the worst. It's a small UI change with an outsized effect on how many people ever reach a results page at all.

## How do I install Nobi's typeahead search on BigCommerce?

Nobi's search component installs on a BigCommerce storefront with one line of embed code added to your Stencil theme, replacing or augmenting the default search box - no engineering project required. Once the script is live, the instant-search index builds from your existing product catalog automatically, so you're not re-entering products anywhere.

There are two ways to add it. The first is a direct install: paste this into your theme's header, right before the closing `</head>` tag, and swap in your merchant ID from the Installation page of your Nobi dashboard.

```html
<script async src="https://assistant-script.nobi.ai/nobi.bundle.js" onload="window.Nobi.initialize({ accountId: 'YOUR_MERCHANT_ID' });">
</script>
```

If your team manages tags through Google Tag Manager instead of editing theme files directly, create a new Custom HTML tag, name it something like "Nobi AI Assistant," paste in this initialization code with your account ID swapped in, set the trigger to All Pages, then save and publish:

```html
<script>
(function () {
 var scriptUrl = 'https://assistant-script.nobi.ai/nobi.bundle.js';

 function initializeNobi() {
 if (!window.Nobi || typeof window.Nobi.initialize !== 'function') {
 return;
 }
 window.Nobi.initialize({
 merchantId: 'YOUR_ACCOUNT_ID',
 autoLaunch: ['floating-chat-bar']
 });
 }

 if (window.Nobi && typeof window.Nobi.initialize === 'function') {
 initializeNobi();
 return;
 }

 var existingScript = document.querySelector('script[src="' + scriptUrl + '"]');
 if (existingScript) {
 existingScript.addEventListener('load', initializeNobi, { once: true });
 return;
 }

 var script = document.createElement('script');
 script.async = true;
 script.src = scriptUrl;
 script.onload = initializeNobi;
 document.head.appendChild(script);
})();
</script>
```

Either path needs a place for the search bar to render. Drop this wherever you want it to show up in your theme:

```html
<nobi-search-bar
 default-mode="ai"
 size="regular"
 show-mode-toggle="false"
 show-hint-row="false">
</nobi-search-bar>
```

Once it's live, open your store, confirm the search bar shows up, and type a partial query to check that results appear before you finish typing. Most teams get through this quickly, without the drawn-out timeline a full search migration usually takes.

## What shows up in the typeahead dropdown as a shopper types?

Here's what actually populates that dropdown. As a shopper types, Nobi's dropdown surfaces matching products, collections, and site content ranked by relevance to the characters entered so far, all before the query is submitted. This is instant matching on what's been typed, not semantic understanding. A shopper who types "wid..." sees products starting with those characters, not products related by meaning to a word they haven't finished typing yet.

Products aren't the only thing that can show up. A collection page (a "Wide-Fit Sandals" PLP, for example) or a content page like a sizing guide can surface in the same dropdown if its title matches the characters typed so far. That matters because a shopper typing "wid" might be looking for a specific product, or they might be trying to find the wide-fit category page itself. Showing both in one dropdown, ranked by how closely each matches the typed characters, means the shopper doesn't have to guess which kind of result they'll get before they've finished typing.

Semantic matching, the part that reads intent and handles a shopper's phrasing rather than their exact characters, only kicks in once that query gets submitted. Typing "wid" doesn't trigger Nobi to reason about what "wide" might mean or pull in items related by concept. It just matches what's on the page against what's been typed so far. That's a deliberate split, not a limitation. Typeahead's job is to keep the shopper company while they type, proving the store is listening in real time. Once they hit enter, or pause long enough to submit a full thought like "wide sandals that don't rub," the query moves to a different kind of matching built to understand what they actually meant. Keeping those two stages separate is what makes each one fast at the job it's actually doing.

## How do I measure the lift from adding typeahead search?

Nobi's search analytics dashboard shows conversion rate, revenue per visitor, and search-to-cart behavior before and after typeahead goes live, so you can attribute the change to the dropdown itself instead of guessing at what moved the number. That matters because typeahead and semantic search are two different mechanisms, tracked separately, and lumping them into one "search got better" metric hides which one actually did the work. On a comparable A/B test, adding Nobi's typeahead to [UNTUCKit](/customers/untuckit)'s storefront lifted conversion rate 0.5 percentage points on an absolute basis, isolated from the semantic-search impact.

The measurement window is simple: pull conversion rate and revenue per visitor for a few weeks before typeahead goes live, then the same window after. Because the dropdown changes behavior at the first keystroke, most of the lift shows up fast, within the first couple of weeks, not months out.

Watch zero-result rate on submitted queries as a separate line from typeahead engagement. These are two different failure points. Typeahead engagement tells you how many shoppers interacted with the dropdown before submitting anything. Zero-result rate tells you how many submitted queries came back empty. A store can fix one and still have a problem with the other: shoppers might be engaging with the dropdown constantly but still landing on empty results pages when they type something the catalog doesn't match. Tracking both keeps you from declaring victory on a metric that only tells half the story.

Nobi's dashboard splits these two mechanisms out at the reporting level, showing instant-search interactions separately from semantic-search interactions. That split is what makes the UNTUCKit number possible to isolate in the first place. Without it, a conversion lift after launch could come from either the dropdown, the relevance matching, or some unrelated seasonal shift, and there'd be no way to tell which lever to pull if the number didn't move the way you expected.

## When does BigCommerce search need more than a typeahead dropdown?

A typeahead dropdown fixes the drop-off problem, but it won't solve every search decision your team faces. If your team wants engineering-level control over ranking logic, or needs a merchandiser-controlled rule engine that can point to a specific setting for every query pattern, a dedicated tool built for that exact job is the better fit.

Algolia is the developer-first option. It's a search API, so ranking rules, synonyms, and merchandising configs all live in code your engineering team writes and maintains. That's the right call when you have a dedicated search engineer who wants to build custom ranking logic instead of relying on a default relevance model, and you're comfortable owning that code long-term.

Searchspring runs the other direction. Every query pattern gets a rule a merchandiser sets by hand, and every result on that query traces back to the rule that produced it. Teams that want that level of manual, query-by-query control over what shows up in the dropdown, and are fine maintaining a growing rule library as the catalog changes, get exactly that with Searchspring.

Nobi covers typeahead, semantic search, and shopper Q&A together, but it isn't either of those tools. There's no code-level API for custom ranking, and there's no rule engine for pinning a specific product to a specific query. Personalization today is limited to placeholder text and starter messages tailored to the page a shopper's on, not results that reorder based on an individual shopper's click or purchase history. If personalization is the main reason you're evaluating search tools, that's worth knowing going in.
